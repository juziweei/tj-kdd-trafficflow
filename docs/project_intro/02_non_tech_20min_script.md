# 20 分钟非技术讲稿（可直接口播）

> 适用对象：导师、跨领域评委、业务方
>
> 目标：让听众清楚你“做了什么、为什么这么做、证据在哪里、风险怎么处理”

---

各位老师好，我今天汇报的项目是 KDD Cup 2017 收费站流量预测。  
我先讲结论：我们在严格不看未来数据的前提下，把主线结果做到 in-split 约 14.95，outer 泛化约 17.60，并且每一步都能复现、能审计。

我希望大家今天听完，不只是记住一个分数，而是记住这件事：我们把“预测模型”做成了一条可持续优化的工程流程。

## 第一部分：我在解决什么问题（约3分钟）

这个任务是预测收费站未来一段时间的车流量。  
单位是 20 分钟一个时间窗，评估看 MAPE，也就是平均误差百分比，数值越低越好。

这件事的业务价值很直接。  
如果预测更准，收费站可以更合理地安排人员，提前发现拥堵风险，节假日应急资源也能更早调度。  
所以它不是“学术上的一个数字游戏”，而是可落地的运营能力。

我在项目开始时，先冻结了三条红线：
- 只用时间切分做验证，不允许随机切分。
- 每次实验必须有 run_id，不能“匿名实验”。
- 提交文件必须通过 schema 校验，不能只看离线分数。

我之所以先做这个，是因为我不希望项目最后变成“看起来很好，但无法复现、无法交付”。

## 第二部分：为什么这样设计（约4分钟）

这个问题有三个本质难点。

第一个难点是非平稳。  
车流在工作日和节假日、早晚高峰与平峰之间差异很大，规则不是固定的。

第二个难点是多步预测误差传播。  
我们不是只预测一个点，而是连续预测多个时间点。前面如果偏一点，后面会被放大。

第三个难点是切片差异。  
不同收费站、不同方向、不同 horizon，误差结构并不一致。一个方案在 A 切片有效，不一定在 B 切片有效。

基于这三个难点，我没有一次性上最复杂模型，而是分阶段推进：
`baseline -> gbdt -> strong backbone -> fusion -> post-process`。

这条路线的逻辑是：
- 先把“流程正确”建立起来；
- 再增加“模型能力”；
- 最后再做“稳健融合和校正”。

这样做的好处是每一步都能解释因果，不会出现“分数变了但不知道为什么”。

## 第三部分：我具体做了什么（约5分钟）

### 阶段 A：Baseline
我先搭了一个可靠基线，让数据处理、特征构建、训练验证、提交导出全部跑通。  
这一步我重点不是冲分，而是确定所有环节遵守时间因果。

### 阶段 B：GBDT 增强
在基线可用后，我引入 GBDT 分支，增强对复杂非线性模式的建模能力。  
同时我做了全局与分组的组合尝试，看看是否能改善关键切片。

### 阶段 C：Strong Backbone v6（当前主线）
我把当前主线统一到 `run_strong_backbone_v6.py`。  
这个阶段是系统化升级：
- 用线性分支做稳态底盘；
- 用双专家分支处理更复杂模式；
- 用融合机制动态组合多个分支；
- 用后处理减少系统性偏差。

### 阶段 D：受控迭代与验收
我们做了 13 轮小步迭代，不是盲调。  
每一轮都要同时过四关：overall、rolling 稳定性、关键切片、副作用护栏。  
最终通过的是第 8 轮 `r8_score_shift`，不是因为它“看起来最激进”，而是因为它“综合最稳”。

### 阶段 E：Outer 泛化
我把验证边界推到更严格的外段，重新评估可迁移性。  
单模型主线是 `17.6376`，随后我们做低自由度固定融合，把结果稳定压到 `17.6014`。  
这里我刻意避免高自由度路由，是为了减少验证集定制风险。

## 第四部分：结果为什么可信（约3分钟）

可信不是一句“我觉得”。  
我们的证据是 run 级链路：每次实验都有 run_id、配置、命令、指标、产物，能完整回放。

核心结果是：
- in-split 稳健主线约 `14.95`
- outer 泛化当前 best 约 `17.60`

同时我们不只看 overall。  
我们还看 horizon 和收费站方向切片，避免“平均分漂亮、局部崩盘”。

还有一点很关键：
我们保留失败实验并写复盘，不只展示成功样本。  
这让结论更可信，也让后续迭代更高效。

## 第五部分：风险与失败（约3分钟）

我会主动讲两个风险。

第一个风险是泛化差距。  
in-split 和 outer 之间仍有差距，说明分布外挑战还没有彻底解决。

第二个风险是复杂系统耦合。  
某些参数改动会引发连锁反应，局部提升可能换来全局波动。

我们已经遇到过这类失败：
- 有的轮次 overall 降了，但 rolling 变差，被拒绝；
- 有的策略在内段很好，外段明显退化；
- 有的独立分支出现递推失稳，说明并非所有“更复杂”都更好。

我把这些失败都当作有效信息，而不是噪音。  
它们帮助我们收敛到了更稳健的路线。

## 第六部分：可复现、可审计、下一步（约2分钟）

在可复现方面，我们已经形成固定机制：
- run_id 强制
- 配置文件化
- 指标与预测落盘
- 提交 schema 校验
- 会话协议持续更新

所以这不是“只能由作者讲清楚”的项目，而是“别人也能复跑验证”的项目。

下一步我会做三件事：
1. 扩大外推验证窗口，优先提升 outer 表现；
2. 针对长 horizon 误差做结构化约束；
3. 控制自由度，继续做小步、可解释、可回滚的优化。

最后用一句话收尾：  
这个项目最大的价值，不是某一个最好分数，而是我们建立了一套在严格约束下持续提分、并且可信可审计的方法。
